# 1. OSI 7계층
**OSI 7 계층이란?** 네트워크 통신 과정을 7단계로 나눈 모델   
**왜 나눴을까?**    
- 복잡한 네트워크를 단계별로 나눠서 이해
- 기능과 역할을 계층별로 분리하여 문제 발생 위치를 빠르게 파악 가능

## 1-1. 각 계층
**7계층 응용(Application)**     
사용자와 직접적으로 연결되는 계층   
**특징** :   
- 사용자가 사용하는 네트워크 서비스 제공 계층
- 다른 계층을 통해 실제 전송이 이루어짐 

**역할** : 
- 웹 요청 생성 (HTTP GET, POST)
- 파일 업로드/다운로드 (FTP)
- 이메일 송수신 (SMTP, IMAP)
- 도메인 -> IP 변환 요청 (DNS)   
  
**단위** : Data     
**EX)** HTTP/HTTPS, FTP

---

**6계층 표현(Presentation)**     
데이터 형식 처리 담당   
**특징** : 
- 서로 다른 시스템 간 데이터 해석 가능하게 만듦
- 보안(암호화)과 데이터 표현 담당

**역할** : 
- 인코딩 / 디코딩 (UTF-8, ASCII)
- 암호화 / 복호화(TLS 기반)
- 압축 / 해제

**EX)** 
- 한글 데이터 깨짐 방지 -> UTF-8
- HTTPS -> TLS를 이용한 암호화 통신
- 이미지 전송 -> JPEG 포맷 유지

---

**5계층 세션(Session)**     
연결 유지 / 관리 담당   
**특징** : 
- 통신의 연결 상태 관리
- 실제로는 TCP 또는 애플리케이션 계층에 포함되는 경우가 많음

**역할** : 
- 세션 생성 / 유지 / 종료
- 통신 중단 시 재연결 지원
- 동기화 포인트 설정

**EX)** 
- 화상 회의 / 영상통화
- 스트리밍 이어보기
- 원격 접속

---

**4계층 전송(Transport)**     
데이터 전달 책임 담당  
**특징** :  
- 포트 번호를 사용해 어떤 응용 프로그램으로 전달할지 구분
- 신뢰성, 속도, 흐름 제어 담당

**역할** : 
- 데이터 분할 (Segment 생성)
- 순서 번호 부여
- 오류 발생 시 재전송
- 흐름 제어 / 혼잡 제어

**단위** : Segment (TCP) / Datagram (UDP)  
**EX)** 
- 웹 (HTTP) -> TCP
- 게임, 스트리밍 -> UDP

---

**3계층 네트워크(Network)**     
라우팅 및 네트워크 간 데이터 전달 담당  
**특징** : 
- 서로 다른 네트워크 간 통신 가능
- 논리 주소(IP 주소) 기반 통신

**역할** : 
- 출발지 / 목적지 IP 설정
- 라우팅 (경로 선택)
- 패킷 전달

**단위** : 패킷(Packet)     
**EX)** 
- IP (IPv4, IPv6)
- ICMP (ping)

---

**2계층 데이터링크(Data Link)**     
같은 네트워크(LAN) 내부에서 데이터를 전달하는 계층  
**특징** : 
- MAC 주소 기반 통신
- 스위치가 핵심 장비

**역할** : 
- 프레임 생성 (MAC 주소 포함)
- 오류 검출 (CRC)
- 같은 LAN 내 전달

**단위** : 프레임(Frame)    
**EX)** 이더넷(Ethernet), ARP

---

**1계층 물리(Physical)**     
데이터를 실제 신호로 변환하여 전송하는 계층 
**특징** : 
- 가장 하위 계층
- 하드웨어 중심

**역할** : 
- 0과 1을 전기/빛 신호로 변환
- 케이블과 전파 사용
   
**단위** : 비트(Bit)   

# 2. TCP/IP
**TCP/IP란?** 인터넷에서 사용하는 대표적인 네트워크 프로토콜 집합   
(컴퓨터끼리 데이터를 주고받기 위한 통신 규칙 모음)  
**왜 중요할까?** 인터넷 표준 통신 구조임    

| TCP/IP | OSI |
| ----- | ----- |
| 응용 계층 | 5~7 계층 |
| 전송 계층 | 4 계층 |
| 인터넷 계층 | 3 계층 |
| 네트워크 엑세스 계층 | 1~2 계층 |

## 2-1. 각 계층
**응용 계층(Application)**      
사용자 프로그램이 네트워크 서비스를 이용하는 계층
| 프로토콜 | 역할 |
| ----- | ----- |
| HTTP/HTTPS | 웹 통신 |
| FTP | 파일 전송 |
| SMTP | 이메일 전송 |
| DNS | 도메인 이름을 IP 주소로 변환 |

**전송 계층(Transport)**    
데이터를 어떻게 안전하고 효율적으로 전달할지 결정하는 계층
- 데이터 분할
- 오류 검사
- 재전송
- 포트 번호 관리

**TCP(Transmission Control Protocol)란?**     
신뢰성 있는 데이터 전송을 제공하는 전송 계층 프로토콜
- 연결 지향 방식
    - 송신자와 수신자가 서로 통신 가능한 상태인지 확인
    - 3-way Handshake 과정 사용
    - 연결 후 데이터 전송 시작
- 신뢰성 보장
    - 패킷 손실 확인 가능
    - 오류 발생 시 재전송 수행
    - ACK(응답 확인) 사용

**UDP(User Datagram Protocol)란?**      
속도를 우선하는 비연결형 전송 프로토콜
- 비연결 지향 방식
    - 연결 과정 없이 바로 데이터 전송
    - 연결 생성 과정 없음
- 신뢰성 보장 못함
    - 데이터 정확성 직접 보장하지 않음

**인터넷 계층(Internet)**       
IP 주소를 이용해 다른 네트워크까지 데이터를 전달하는 계층
- IP 주소 사용
- 경로 선택(라우팅)
- 패킷 전달

| 프로토콜 | 역할 |
| ----- | ----- |
| IP | 주소 지정 |
| ICMP | 오류/진단 |
| ARP | LAN 내부에서 IP를 MAC 주소로 변환 |

**네트워크 엑세스 계층**   
실제 물리 전송 담당
- MAC 주소 사용
- 프레임 생성
- 전기/빛 신호 변환

| 기술 | 설명 |
| ----- | ----- |
| Ethernet | LAN 통신 |
| Wi-Fi | 무선 LAN |
| MAC Address | 물리 주소 |

## 2-2. 특징
-  TCP/IP는 네트워크 기능을 계층별로 분리함 
    - 위/아래 계층과 협력
    - 각 계층이 자기 역할만 수행
    - **왜 나눌까?** 유지보수 및 기술 교체가 쉽고 문제 파악이 빠름
- 패킷 기반 통신 방식   
    - 큰 데이터를 한 번에 보내면 오류 시 전체 재전송 필요   
    -> 따라서 데이터를 작은 패킷 단위로 나누어 전송함
    - 네트워크 부담 증가를 방지함
- IP 기반 주소 체계 사용
    - 인터넷 전체에서 장치 위치 구분 가능
    - IP 주소로 출발지, 목적지 등을 식별 가능

## 2-3. 장점
- 서로 다른 장비와 운영체제 간 통신 가능
    - **왜 가능할까?** 프로토콜이 표준화되어 있기 때문
    - 장비 선택이 자유로움
- 인터넷 규모 확장 가능
    - 작은 LAN부터 인터넷 전체까지 사용 가능
    - **왜 가능할까?** IP 기반 라우팅 구조 때문
- 패킷 교환 방식으로 효율적
    - 패킷 손실 시 일부만 재전송 가능
    - 네트워크 자원 효율적 사용

## 2-4. 단점
- 구조가 복잡함
    - **포함 요소 예시** : IP, TCP, UDP, ARP, DNS, HTTP
    - 초보자 입장에서 이해하기 어려움
- 헤더 오버헤드 발생
    - 실제 데이터보다 제어 정보 증가 가능   
    -> 대역폭 사용 증가 / 작은 데이터 전송 비효율
- 기본 보안 기능 부족
    - **이유?** 암호화가 없고 인증이 부족하기 때문에
    - IP Spoofing, ARP Spoofing 등의 공격에 취약
- 브로드캐스트 문제 발생 가능
    - LAN 전체 장비가 브로드캐스트 패킷을 처리해야 하기 때문에 트래픽 및 CPU 부담 증가

# 3. Subnet
**서브넷(Subnet)이란?** 하나의 큰 네트워크를 여러 개의 작은 네트워크로 나누는 것    

**왜 나눌까?** 큰 네트워크 하나로 모든 장비를 연결하면 비효율적이기 때문에

**특징**    
- 큰 네트워크를 작은 네트워크로 분리    
    - 하나의 큰 LAN을 여러 개의 논리적 네트워크로 나눔
    - 부서/용도별 네트워크 분리 가능
    - 네트워크 혼잡 감소 효과 존재
- 네트워크 영역과 호스트 영역 구분
    - IP 주소를 네트워크 부분과 호스트 부분으로 나눠 사용
    - 네트워크 주소 / 브로드캐스트 주소 계산 가능
- 라우터를 통한 네트워크 간 통신
    - 서로 다른 서브넷끼리는 직접 통신 불가능
    - 라우터가 목적지 네트워크를 판단 후 전달
    - 서로 다른 VLAN 간 통신에도 사용

**장점**    
- 브로드캐스트 트래픽 감소
    - 브로드캐스트가 같은 서브넷 내부로 제한됨
    - ARP 패킷 범위 감소
    - 네트워크 혼잡 완화 가능
- 네트워크 성능 향상
    - 장비가 처리해야 하는 트래픽 감소
    - 스위치 및 CPU 부담 감소
    - 대규모 네트워크 환경에서 효율적
- IP 주소 효율적 사용
    - 필요한 수만큼 IP 할당 가능
    - IPv4 주소 부족 문제 완화 가능

**단점**    
- 설계가 복잡함
    - 서브넷 마스크 계산 필요
    - 잘못 설계하면 IP 낭비 발생 가능
- 관리 어려움 증가 가능
    - 서브넷 수 많아질수록 관리 포인트 증가
    - VLAN/라우팅 정책 함께 관리 필요
    - 유지보수 복잡도 증가 가능
- 라우터 부담 증가
    - 서브넷 간 통신 시 라우터 반드시 필요
    - 패킷 처리량 증가 가능
    - 장비 성능 요구 증가
- 잘못 설정 시 통신 문제 발생
    - 서브넷 마스크 오류 시 통신 실패 가능
    - 게이트웨이 설정 오류 발생 가능
    - IP 충돌 문제 발생 가능

## 3-1. Subnet Mask
**서브넷 마스크(Subnet Mask)란?** IP 주소에서 네트워크 부분과 호스트 부분을 구분해주는 값   

**왜 필요할까?** IP 주소 중 어디까지가 네트워크인지 구분해줌

**역할**    
- 같은 네트워크인지 판단
- 네트워크 주소 계산
- 브로드캐스트 주소 계산
- 라우팅 판단

## 3-2. Cast
**캐스트(Cast)란?** 네트워크에서 데이터를 누구에게 보내는지에 대한 전송 방식

**유니캐스트(Unicast)**    
한 장치한테만 보내는 1:1 방식   
- 특정 장치만 데이터 수신
    - 스위치가 목적지 MAC 주소로 확인 후 해당 포트로만 전달
    - 불필요한 장비는 패킷 처리 안 함
- 연결 대상이 명확함
    - 송신자와 수신자 구분 쉬움
    - 트래픽 분석 및 문제 추적 쉬움
    - 패킷 흐름 관리 쉬움   

장점
- 불필요한 장비가 패킷 처리하지 않음
    - 네트워크 효율 증가
    - CPU 부담 감소
    - 트래픽 낭비 감소
- 보안성이 상대적으로 높음
    - 특정 장치만 데이터 수신
    - 데이터 노출 범위 제한 가능

단점
- 동일 데이터 반복 전송 필요
    - 같은 데이터를 여러 사용자에게 보낼 경우 각각 따로 전송
    - 대역폭 사용량 증가
- 연결 수 증가 시 성능 저하 가능
    - 사용자 많을수록 세션 증가
    - 메모리 및 CPU 사용량 증가

**브로드캐스트(Broadcast)**     
네트워크 전체에게 보내는 방식   
- 목적지 MAC 주소를 모를 때 사용
    - 네트워크 초기 탐색
    - ARP Request
    - DHCP Discover
- 라우터는 기본적으로 차단
    - 브로드캐스트는 다른 네트워크로 전달 안 됨
    - Broadcast Domain 내부에서만 동작

장점
- 자동 탐색 기능 가능
    - 네트워크 장비 자동 발견
    - DHCP 자동 IP 할당 가능
- 관리 편의성 증가 가능
    - 여러 장비 동시 공지 가능
    - 네트워크 초기 설정 유리

단점
- 모든 장비가 패킷 처리해야 함
    - CPU 사용량 증가
    - 불필요한 처리 발생
- 네트워크 혼잡 증가 가능
    - 장비 수 많아질수록 브로드캐스트 증가
    - 대역폭 낭비 가능

**애니캐스트(Anycast)**     
여러 서버 중 가장 가까운 하나에게 보내는 방식   
- 라우팅 기반 동작
    - BGP 같은 라우팅 프로토콜 사용 가능
    - 최적 경로 서버 자동 선택 가능
- 응답 속도 최적화 가능
    - 가까운 서버 연결
    - 지연 시간 감소 가능
- 부하 분산 가능
    - 여러 서버로 사용자 분산
    - 특정 서버 과부화 감소

장점
- 서버 부하 분산 가능
    - 트래픽 여러 서버 분산
    - 서비스 안정성 향상
- 글로벌 서비스 운영 효율적
    - 국가별 최적 서버 연결 가능
- 장애 대응 가능
    - 특정 서버 장애 시 다른 서버 연결 가능
    - 가용성 증가

단점
- 구현 및 운영 복잡
    - BGP 설정 필요 가능
    - 글로벌 라우팅 이해 필요
- 데이터 동기화 문제 가능
    - 여러 서버 데이터 일치 필요
    - 캐시 불일치 가능

**멀티캐스트(Multicast)**   
필요한 사람한테만 한 번에 보내는 방식   
- 동일 데이터 대량 전송에 최적화
    - 한 번 전송으로 여러 사용자 전달 가능
    - 스트리밍 환경에 적합
- 대역폭 효율 높음
    - 유니캐스트처럼 여러 번 전송 안 함
    - 네트워크 자원 절약 가능

장점
- 서버 부담 감소
    - 사용자 수 증가해도 반복 전송 감소
    - CPU 및 메모리 사용량 절감 가능
- 대역폭 절약 가능
    - 동일 데이터 한 번만 전송
    - 네트워크 사용량 감소
- 실시간 서비스에 효율적
    - ex) IPTV, 실시간 방송, 화상회의

단점
- 설정 및 관리 복잡
    - 그룹 관리 필요
    - 가입/탈퇴 관리 필요
    - 네트워크 설계 어려움 가능
- 장비 지원 필요
    - 스위치/라우터 멀티캐스트 지원 필요
    - IGMP Snooping 필요 가능
- 장애 분석 어려움
    - 그룹 기반 구조라 트래픽 추적 복잡 가능

## 3-3. Subnetting
**서브넷팅(Subnetting)이란?** 하나의 큰 네트워크를 여러 개의 작은 네트워크로 나누는 기술    
(큰 LAN을 여러 개의 작은 LAN처럼 사용하는 것)

**왜 사용할까?**    
- 브로드캐스트 트래픽 감소시키기 위해
- 네트워크 성능 향상시키기 위해
- IP 주소 효율적 사용하기 위해
- 네트워크 관리 편리성을 높이기 위해

**FLSM(Fixed Length Subnet Mask)**      
동일한 크기의 서브넷 마스크 사용하는 고정 길이 서브넷 마스크 방식   
- 모든 서브넷이 동일한 서브넷 마스크 사용
    - 네트워크를 나눌 때 모든 서브넷 크기를 똑같이 설정
    - 각 서브넷의 호스트 수 동일
    - 모든 네트워크 구조가 일정함
- 호스트 수가 전부 동일
    - 각 부서/네트워크가 같은 개수의 IP 사용 가능
    - 네트워크 설계가 단순해짐
    - 작은 부서는 IP 낭비 가능성 존재
- 네트워크 확장 유연성 낮음
    - 부서별 요구 IP 수 달라도 동일 크기 유지
    - 중간 크기 조절 어려움
    - 구조 변경 시 재설계 가능성 존재

**VLSM(Variable Length Subnet Mask)**   
다른 크기의 서브넷 마스크 사용하는 가변 길이 서브넷 마스크 방식     
- 서브넷마다 다른 크기의 서브넷 마스크 사용 가능
    - 필요한 호스트 수에 맞춰 네트워크 크기 조절 가능
    - 큰 부서는 큰 서브넷 사용
    - 작은 부서는 작은 서브넷 사용 가능
- IP 주소를 매우 효율적으로 사용 가능
    - 불필요한 IP 낭비 감소
    - 필요한 만큼만 할당 가능
    - IPv4 주소 부족 문제 완화 가능
- 네트워크 설계 유연성 높음
    - 다양한 크기 네트워크 혼합 가능
    - 환경 변화 대응 쉬움
    - 네트워크 확장 유리

| 항목 | FLSM | VLSM |
| ----- | ----- | ----- |
| 서브넷 크기 | 모두 동일 | 서로 다름 |
| IP 효율 | 낮음 | 높음 |
| 계산 난이도 | 쉬움 | 어려움 |
| 설계 유연성 | 낮음 | 높음 |
| 관리 난이도 | 단순 | 복잡 가능 |

# 4. DHCP
**DHCP(Dynamic Host Configuration Protocol)란?** 장치가 네트워크에 접속할 때 IP 주소와 네트워크 설정을 자동으로 할당해주는 프로토콜     
(IP 자동 설정 시스템)   
- 자동 IP 할당 가능
    - 사용자 직접 설정 불필요
    - 네트워크 연결 즉시 사용 가능
- 중앙 집중 관리 가능
    - DHCP 서버 하나로 여러 장치 관리 가능
    - IP 중복 관리 쉬움
    - 설정 변경 쉬움
- 브로드캐스트 기반 초기 통신 사용
    - 처음에는 서버 IP 모름
    - DHCP Discover 브로드캐스트 사용

장점
- 네트워크 설정 자동 제공 가능
    - 사용자가 네트워크 지식 없어도 사용 가능
    - 연결 즉시 인터넷 사용 가능
    - 설정 오류 감소 가능
- 이동 환경에 적합
    - 네트워크 바뀔 때마다 자동으로 새로운 IP 받기 가능
    - 모바일 환경에 적합

단점
- DHCP 서버 장애 시 문제 발생
    - 자동 네트워크 설정 실패
    - 인터넷 사용 어려움 발생 가능
    - 중앙 서버 의존성 존재
- 브로드캐스트 트래픽 발생
    - 같은 LAN 내부 장비들이 브로드캐스트 패킷을 수신
    - 장비 수 많아질수록 부담 증가
- 특정 장비는 고정 IP 필요
    - 일부 장비는 주소 변경되면 문제 발생   
    ex) 서버, 프린터, NAS, CCTV 등
    - DHCP 재할당 시 IP 변경 가능

**왜 필요할까?**    
- 네트워크 설정 자동화 가능하게 하기 위해
- IP 주소 중복 방지 가능하게 하기 위해
- 대규모 네트워크 관리를 쉽게 하기 위해

**동작 과정(DORA 과정)**    
**D**iscover : 클라이언트가 DHCP 서버를 찾는 단계   
**O**ffer : DHCP 서버가 사용 가능한 IP를 제안하는 단계  
**R**equest : 클라이언트가 특정 DHCP 서버의 제안을 선택하는 단계    
**A**CK : DHCP 서버가 최종 승인하는 단계

# 5. CIDR 표기법
**CIDR(Classless Inter-Domain Routing)란?** IP 주소를 더 효율적으로 사용하기 위해 만든 주소 표기 및 라우팅 방식     
(IP 주소 뒤에 /숫자를 붙여서 네트워크 크기를 표현하는 방식)  

**/숫자** = 네트워크 영역으로 사용하는 비트 수
- IPv4는 총 32비트
- 앞에서 몇 비트를 네트워크로 쓸지 정함
- ex) 192.168.0.1/24
    - 앞 24비트 = 네트워크
    - 뒤 8비트 = 호스트

**왜 필요할까?** 
- IPv4 주소 낭비 문제를 해결하기 위해
- 네트워크를 유연하게 설계하기 위해
- 라우팅 효율 향상을 위해

# 6. ARP/RARP
**ARP와 RARP란?** IP <-> MAC 주소 변환 프로토콜     
(OSI 2계층 + 3계층(IP) 사이에서 작동 / 브로드캐스트 기반 요청 사용)

**공통점**  
- LAN 기반으로 동작함
    - 같은 네트워크 안에서만 동작
    - 라우터 넘어가면 사용 불가     
    -> 인터넷 전체용 프로토콜 아님
- 브로드캐스트 사용함
- MAC 주소를 기반으로 동작
    - ARP : MAC 찾아냄
    - RARP : MAC 기반으로 IP 찾음
- IP 통신을 가능하게 하기 위한 보조 프로토콜임
    - 단독으로 인터넷을 제공하는 게 아님
    - TCP/IP 통신을 가능하게 만드는 도우미 역할

## 6-1. ARP
**ARP(Address Resolution Protocol)란?** IP 주소를 MAC 주소로 변환하는 프로토콜   
(같은 네트워크인 LAN 안에서 데이터를 전송하기 위해 사용됨)      
**왜 필요할까?** IP만 알아서는 실제 전송이 불가능해서

**동작 과정**   
- **ARP Request (브로드캐스트)** : 송신 PC가 네트워크 전체에 질문함
    - 브로드캐스트 방식 사용
    - 같은 LAN 전체에 뿌림
- **ARP Reply (유니캐스트)** : 해당 IP를 가진 장치가 응답함
    - 유니캐스트 방식
    - 요청한 장치에게만 응답
- **ARP Cache 저장** : 송신 PC가 결과 저장  
    - 브로드캐스트 트래픽 감소 및 통신 속도 향상 가능
    - 일정 시간 후 자동 삭제되어 최신 네트워크 상태를 유지
- **실제 데이터 전송** : 이제 MAC 주소를 알게 됐으므로
    - Ethernet Frame 생성
    - 목적지 MAC 포함
    - 스위치가 전달

**특징**    
- IP 주소를 MAC 주소로 변환함
    - 실제 LAN 통신을 가능하게 함
    - 논리 주소(IP)를 물리 주소(MAC)로 변환하는 프로토콜
- 같은 네트워크 안에서만 동작
    - ARP는 로컬 네트워크 내부에서 사용됨
    - 다른 네트워크 장치의 MAC 주소는 직접 찾지 못함
- 브로드캐스트 기반 동작
    - 대상 장치만 응답함
    - 같은 LAN에 있는 모든 장비가 요청을 받음
- ARP Cache에 저장해서 재사용
    - 한 번 찾은 MAC 주소는 메모리에 저장
    - 빠른 통신 가능
    - 일정 시간이 지나면 자동 삭제됨 (속도 향상 및 TTL 존재)

**장점**    
- IP만 알아도 실제 통신 가능
    - ARP 자체가 IP를 MAC주소로 변환을 하기 때문에 연결이 쉬움
- 자동으로 MAC 주소 탐색 가능
    - 사용자 편의성 증가
    - 네트워크 자동화 가능
    - 관리 쉬움
- ARP Cache로 성능 향상
    - 한 번 찾은 MAC 주소를 저장하기 때문에 다시 ARP 안 함      
    -> 바로 Frame 생성 가능
- 브로드캐스트 감소 효과
    - 이미 저장된 MAC를 재사용하기 때문에   
    -> 네트워크 혼잡 감소       
    -> 불필요한 패킷 감소

**단점**        
- 브로드캐스트 트래픽 발생
    - ARP는 LAN 전체에 전송됨   
    -> 네트워크 장비 전부 메시지를 받음
    - 장치가 많아질수록 브로드캐스트 증가
    - CPU 사용량 증가
- 보안 취약점 존재
    - ARP는 인증 기능이 없기 때문
    - 실제 피해 예시 ) 로그인 정보 탈취, 패킷 감청, 세션 하이재킹
- LAN 내부에서만 동작 가능
    - 같은 네트워크 안에서만 사용 가능
    - 라우터를 넘어가면 목적지 서버 MAC이 아닌 기본 게이트웨이 MAC만 찾음       
    -> 인터넷 전체 탐색 불가능
- 오래된 캐시 정보 문제
    - ARP Cache는 임시 저장     
    -> Cache에 예전 정보가 남아있으면 잘못된 장치로 전송 가능

## 6-2. RARP
**RARP(Reverse Address Resolution Protocol)란?** MAC 주소를 이용해서 자신의 IP 주소를 알아내는 프로토콜     
**왜 필요할까?** 부팅 직후 IP 주소를 자동으로 얻기 위해

**동작 과정**     
- **클라이언트 부팅**   
컴퓨터 전원을 킴
    - MAC 주소는 하드웨어에 저장되어 있음   
    -> 하지만 IP 주소 없음 / 네트워크 설정 없음
    - MAC 기반 통신 일부 가능   
    - IP 기반 TCP/IP 통신 불가능

- **RARP Request 생성**       
클라이언트가 자신의 IP 주소를 요청하는 패킷 생성  
    - 출발지 MAC -> 자기 MAC 사용 가능
    - 목적지 MAC -> 브로드캐스트 사용
    - **왜 사용할까?** 현재 IP 주소를 모르기 때문에 특정 서버 위치를 지정할 수 없음     
    -> 네트워크 전체에 요청 전송해야 함
- **네트워크 전체로 브로드캐스트**   
    - 스위치가 모든 포트로 프레임 전달
    - LAN 내부 장비들이 전부 받음
- **RARP 서버가 요청 수신**     
RARP 서버가 브로드캐스트 프레임 확인
    - 요청 패킷 내부 MAC 주소 확인
    - 자신의 RARP 테이블 검색
    - 일치하는 정보 발견 시 응답 준비
- **RARP Reply 생성 및 응답**   
RARP 서버가 클라이언트에게 응답 전송
    - 보통 유니캐스트 방식 사용
    - 요청한 클라이언트에게만 전송
    - 네트워크 전체가 아니라 특정 장치만 받음
- 클라이언트 IP 주소 설정       
클라이언트가 RARP Reply 수신 후 IP 설정     
    - TCP/IP 통신 가능
    - 서버 접속 가능
    - 인터넷 사용 가능 상태가 됨

**특징**    
- MAC 주소를 이용해 IP 주소를 알아냄
    - 클라이언트는 자기 MAC 주소만 알고 있음    
    -> 서버가 MAC에 대응되는 IP 주소 알려줌
- 부팅 초기 단계에서 사용됨
    - MAC 주소는 NIC(랜카드)에 저장되어 있음
    - 전원 켜자마자 사용 가능
- RARP 서버가 반드시 필요함
    - MAC 주소와 IP 주소 매핑 정보를 저장한 서버 필요
    - 서버 의존적 구조
- IP 주소만 제공 가능
    - 게이트웨이, DNS, 서브넷 마스크 등은 제공 불가능   
    -> **왜?** 구조가 너무 단순해서 IP 주소 하나만 알려주도록 만들어졌기 때문   

**장점**    
- 디스크 없는 장비도 네트워크 사용 가능
    - MAC 주소 기반으로 IP 자동 요청 가능
    - 부팅 직후 네트워크 사용 가능
- IP 주소 자동 설정 가능
    - 설정 시간 감소
    - 초기 네트워크 구성 쉬움
    - 사용자 편의성 증가
- 중앙 집중 관리 가능
    - 관리자가 한 곳에서 통제 가능
    - IP 중복 관리 쉬움
    - 장비 식별 가능
- 구조가 단순함
    - 초기 네트워크 환경에 적합
    - 장비 부담 적음    
**단점**    
- IP 주소만 제공 가능
    - IP는 받아도 인터넷 사용 제한
    - 추가 설정 필요함
- RARP 서버 반드시 필요함   
클라이언트 혼자 해결 불가능
    - 중앙 서버 의존성 높음
    - 서버 장애 시 IP 할당 불가
- 브로드캐스트 트래픽 발생
    - 장비가 많아질수록 트래픽 증가
    - 성능 저하 가능
    - LAN 내부 장비들이 브로드캐스트 요청을 수신
